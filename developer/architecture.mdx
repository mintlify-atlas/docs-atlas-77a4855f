## Technology stack

Stremio Web is a React single-page application bundled with Webpack. The key technologies are:

<CardGroup cols={2}>
  <Card title="React 18" icon="react">
    UI layer using functional components, hooks, and `React.StrictMode`. No class components.
  </Card>
  <Card title="Less + CSS Modules" icon="palette">
    Styles are written in Less and processed through CSS Modules with a `[local]-[hash:base64:5]` naming strategy.
  </Card>
  <Card title="TypeScript" icon="code">
    New components are written in TypeScript (`.ts`/`.tsx`). Legacy components use CommonJS `.js` files. Both are compiled in the same Webpack pipeline.
  </Card>
  <Card title="Webpack 5" icon="box">
    Bundles the app with Babel for JS, `ts-loader` for TypeScript, `less-loader` + `css-loader` for styles, and Workbox for the service worker.
  </Card>
  <Card title="i18next" icon="globe">
    Internationalisation via `i18next` and `react-i18next`. The interface language is driven by the user's profile settings stored in stremio-core.
  </Card>
  <Card title="Workbox" icon="shield">
    Progressive Web App support. In production builds, `WorkboxPlugin.GenerateSW` generates a service worker that pre-caches all assets up to 20 MB.
  </Card>
</CardGroup>

## stremio-core-web integration

`@stremio/stremio-core-web` is the Rust-compiled WASM core of Stremio. It runs in a dedicated **Web Worker** so heavy computation never blocks the UI thread.

The `Core` service initialises a `CoreTransport` which spawns the worker and communicates with it via a `Bridge`:

```js
// src/services/Core/CoreTransport.js
const Bridge = require('@stremio/stremio-core-web/bridge');

function CoreTransport(args) {
    const worker = new Worker(`${process.env.COMMIT_HASH}/scripts/worker.js`);
    const bridge = new Bridge(window, worker);

    bridge.call(['init'], [args])
        .then(() => events.emit('init'))
        .catch((error) => events.emit('error', error));

    this.dispatch = async function(action, field) {
        return bridge.call(['dispatch'], [action, field, location.hash]);
    };
    this.getState = async function(field) {
        return bridge.call(['getState'], [field]);
    };
    // ...
}
```

The core emits `NewState` events whenever its internal models change. Components subscribe to specific models through `useModelState`.

## State management

Stremio Web does **not** use Redux or any third-party state manager. All state lives in stremio-core-web and is surfaced to React through two hooks:

### `useModelState`

The primary data-fetching hook. It dispatches a load action, subscribes to `NewState` events for the named model, and returns the current state. State updates are throttled and deep-compared to prevent unnecessary re-renders.

```js
// src/common/useModelState.js
const useModelState = ({ action, model, map, timeout, deps }) => {
    const { core } = useServices();
    const routeFocused = useRouteFocused();
    // ...
    // Subscribes to core NewState events only when the route is focused
    React.useInsertionEffect(() => {
        const onNewState = async (models) => {
            const state = await core.transport.getState(model);
            setState(typeof map === 'function' ? map(state) : state);
        };
        if (routeFocused) {
            core.transport.on('NewState', onNewState);
        }
        return () => core.transport.off('NewState', onNewState);
    }, [routeFocused]);
};
```

Specialised hooks built on `useModelState`:

| Hook | Model | Returns |
|---|---|---|
| `useProfile` | `ctx` | User profile and settings |
| `useNotifications` | `ctx` | Notification counts |
| `useStreamingServer` | `streaming_server` | Streaming server status |

### Route-focused updates

Subscriptions are **only active while the route is the topmost view**. The `RouteFocusedContext` provides a boolean that `useModelState` checks before subscribing to `NewState`. This prevents background routes from re-rendering unnecessarily.

## Build pipeline

The Webpack config (`webpack.config.js`) has two entry points:

| Entry | Output | Description |
|---|---|---|
| `./src/index.js` | `<COMMIT_HASH>/scripts/main.js` | React application |
| `@stremio/stremio-core-web/worker.js` | `<COMMIT_HASH>/scripts/worker.js` | Core WASM worker |

Output filenames are prefixed with the current git commit hash, which acts as a cache-busting mechanism and allows the service worker to serve the correct version.

The Less pipeline is: `less-loader` → `postcss-loader` (cssnano with autoprefixer) → `css-loader` (CSS Modules) → `MiniCssExtractPlugin`.

A **thread-loader** runs Babel, `ts-loader`, `css-loader`, `postcss-loader`, and `less-loader` in worker threads equal to the number of CPU cores, significantly reducing build time on multi-core machines.

## PWA architecture

In production, Workbox generates a service worker using `GenerateSW` with `clientsClaim: true` and `skipWaiting: true`, which means a newly installed service worker takes over immediately without waiting for all tabs to close.

All static assets (JS, CSS, WASM, images, fonts) are pre-cached. The maximum file size cached is **20 MB** to accommodate the WASM binary.

A `manifest.json` and favicons are copied into the build output by `CopyWebpackPlugin`. The dev server does **not** generate a service worker (`SERVICE_WORKER_DISABLED` defaults to `false` but `WorkboxPlugin` is only added in production mode).
